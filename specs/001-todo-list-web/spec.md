# Feature Specification: TODO List Web Application

**Feature Branch**: `001-todo-list-web`
**Created**: 2026-04-12
**Status**: Draft
**Input**: User description: "Создай web приложение, которое реализует простой TODO List. Никакой авторизации или ролей. Пользователь должен видеть список задач, создавать, редактировать, помечать выполненными и архивировать задачи. Задача должна иметь заголовок и описание. Приложение должно быть максимально простым и открываться локально в браузере."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Просмотр и управление задачами (Priority: P1)

Пользователь открывает приложение в браузере и видит список своих активных задач. Каждая задача отображается с заголовком и кратким описанием. Пользователь может отметить задачу как выполненную прямо из списка.

**Why this priority**: Это основной сценарий использования — без него приложение не имеет смысла.

**Independent Test**: Пользователь может открыть приложение, увидеть пустой или заполненный список задач, отметить задачу как выполненную.

**Acceptance Scenarios**:

1. **Given** пользователь открыл приложение, **When** задач нет, **Then** отображается сообщение о том, что задач пока нет
2. **Given** в списке есть задачи, **When** пользователь открывает страницу, **Then** отображаются все активные (невыполненные и неархивированные) задачи с заголовком и описанием
3. **Given** есть активная задача, **When** пользователь нажимает «Выполнено», **Then** задача помечается как выполненная и скрывается из активного списка

---

### User Story 2 - Создание новой задачи (Priority: P1)

Пользователь создаёт новую задачу, указывая заголовок и описание. После сохранения задача появляется в списке активных.

**Why this priority**: Без возможности создания задач приложение не может быть использовано.

**Independent Test**: Пользователь заполняет форму создания задачи, сохраняет — задача появляется в списке.

**Acceptance Scenarios**:

1. **Given** пользователь на странице приложения, **When** вводит заголовок и описание, нажимает «Создать», **Then** задача появляется в списке активных
2. **Given** пользователь пытается создать задачу, **When** заголовок пустой, **Then** выводится сообщение об ошибке, задача не создаётся
3. **Given** пользователь создаёт задачу, **When** описание оставлено пустым, **Then** задача создаётся успешно (описание необязательно)

---

### User Story 3 - Редактирование задачи (Priority: P2)

Пользователь может изменить заголовок и/или описание существующей задачи. Изменения сохраняются и отображаются в списке.

**Why this priority**: Позволяет корректировать данные, но не блокирует базовое использование.

**Independent Test**: Пользователь редактирует существующую задачу — изменения сохраняются и видны в списке.

**Acceptance Scenarios**:

1. **Given** есть задача в списке, **When** пользователь изменяет заголовок и сохраняет, **Then** обновлённый заголовок отображается в списке
2. **Given** есть задача, **When** пользователь пытается очистить заголовок при редактировании, **Then** выводится ошибка, изменения не сохраняются

---

### User Story 4 - Архивирование задач (Priority: P3)

Пользователь может отправить задачу в архив. Архивные задачи скрываются из основного списка, но могут быть просмотрены отдельно.

**Why this priority**: Полезно для организации, но не критично для базовой работы приложения.

**Independent Test**: Пользователь архивирует задачу — задача исчезает из активного списка и появляется в архиве.

**Acceptance Scenarios**:

1. **Given** есть задача в активном списке, **When** пользователь нажимает «Архивировать», **Then** задача перемещается в архив и скрывается из основного списка
2. **Given** у пользователя есть архивные задачи, **When** он открывает раздел архива, **Then** отображаются все архивированные задачи
3. **Given** задача в архиве, **When** пользователь нажимает «Восстановить», **Then** задача возвращается в активный список

---

### Edge Cases

- Как система реагирует на заголовок с одними пробелами? → Считать пустым, показать ошибку
- Что происходит при очень длинном заголовке или описании? → Отображение обрезается в списке, полный текст виден при просмотре/редактировании
- Как обрабатываются специальные символы (HTML-теги) в заголовке/описании? → Экранируются для безопасности (XSS-защита)
- Что если пользователь случайно закрыл браузер во время редактирования? → Данные формы не сохраняются (приложение простое, localStorage не используется)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА отображать список активных (невыполненных и неархивированных) задач
- **FR-002**: Каждая задача ДОЛЖНА иметь заголовок (обязательный) и описание (необязательный)
- **FR-003**: Пользователь ДОЛЖЕН иметь возможность создать новую задачу
- **FR-004**: Пользователь ДОЛЖЕН иметь возможность редактировать заголовок и описание существующей задачи
- **FR-005**: Пользователь ДОЛЖЕН иметь возможность пометить задачу как выполненную
- **FR-006**: Пользователь ДОЛЖЕН иметь возможность архивировать задачу
- **FR-007**: Архивные задачи ДОЛЖНЫ быть доступны для просмотра в отдельном разделе
- **FR-008**: Пользователь ДОЛЖЕН иметь возможность восстановить задачу из архива
- **FR-009**: Система ДОЛЖНА валидировать, что заголовок задачи не пустой и не состоит только из пробелов
- **FR-010**: Система ДОЛЖНА экранировать HTML-сущности в заголовке и описании для предотвращения XSS
- **FR-011**: Задача МОЖЕТ быть создана с пустым описанием
- **FR-012**: Система ДОЛЖНА хранить данные локально в браузере (localStorage)

### Key Entities

- **Задача (Task)**: Единица работы в системе. Атрибуты: заголовок (строка, обязательный), описание (строка, необязательный), статус (новая/выполненная/архивированная), дата создания, дата последнего изменения.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь может создать задачу с заголовком и описанием за менее чем 10 секунд
- **SC-002**: Все основные операции (создание, редактирование, отметка выполненной, архивирование) выполняются без перезагрузки страницы
- **SC-003**: Приложение открывается локально в браузере без необходимости установки дополнительных зависимостей на стороне пользователя
- **SC-004**: Пользователь может独立完成ть все 4 ключевых действия (создать, редактировать, выполнить, архивировать) без дополнительных инструкций
- **SC-005**: Данные задач сохраняются между сессиями браузера (через localStorage)

## Assumptions

- Приложение предназначено для одного пользователя (локальное использование, многопользовательский режим не требуется)
- Данные хранятся только в localStorage браузера — серверная часть отсутствует
- Поддерживаемые браузеры: последние 2 версии Chrome, Firefox, Edge, Safari
- Мобильная поддержка не является обязательной для первой версии (responsive дизайн желателен, но не критичен)
- Поиск и фильтрация задач не требуются для данной версии
- Экспорт/импорт задач не требуется для данной версии
