# Todo List with AI — Конституция проекта

## Core Principles

### I. Современные стандарты разработки

Система ДОЛЖНА соответствовать современным стандартам разработки для выбранного технологического стека.
Это включает:

- Актуальные версии языков, фреймворков и инструментов (без устаревших мажорных версий с EOL)
- Общепринятые паттерны и лучшие практики (SOLID, DRY, KISS, YAGNI)
- Автоматизацию рутинных операций: линтинг, форматирование, CI/CD
- Безопасность по умолчанию: валидация входных данных, защита от инъекций, безопасное хранение секретов

**Обоснование**: Следование современным стандартам снижает технический долг, упрощает поддержку и привлекает разработчиков.

### II. TDD и покрытие тестами (НЕОТМЕНИМО)

Система ДОЛЖНА быть покрыта тестами. Предпочтителен подход TDD (Test-Driven Development):

- Тесты пишутся ДО реализации функциональности
- Цикл Red-Green-Refactor строго соблюдается: тест написан → тест падает → реализация → рефакторинг
- Минимальное покрытие кода тестами: 80% для бизнес-логики, 100% для критических путей
- Тесты ДОЛЖНЫ быть независимыми, детерминированными и быстрыми
- Типы тестов: unit-тесты, интеграционные тесты, contract-тесты, e2e-тесты (где применимо)

**Обоснование**: TDD гарантирует, что код проектируется с учётом тестируемости, а высокое покрытие снижает риск регрессий.

### III. Читаемость и понятность кода

Код ДОЛЖЕН быть читаемым и понятным для любого разработчика, знакомого с технологическим стеком:

- Осмысленные имена переменных, функций, классов (раскрывающие намерение)
- Функции и методы ДОЛЖНЫ иметь одну ответственность (SRP)
- Максимальный размер функции: 30 строк (рекомендация), жёсткий лимит: 50 строк
- Сложная бизнес-логика ДОЛЖНА сопровождаться комментариями, объясняющими «почему», а не «что»
- Запрещён «мёртвый» код: неиспользуемые импорты, переменные, закомментированные фрагменты

**Обоснование**: Читаемый код снижает когнитивную нагрузку, ускоряет онбординг и уменьшает количество ошибок при модификациях.

### IV. Русский язык кодовой базы

Все комментарии, документация, сообщения коммитов, названия веток (в части описания) и тексты ошибок ДОЛЖНЫ быть на русском языке:

- Комментарии в коде — на русском
- README, CONTRIBUTING, архитектурные документы — на русском
- Сообщения коммитов — на русском (допускается английская терминология)
- Исключения: имена переменных/функций/классов — на английском (общепринятая практика)
- Документация API и контрактов — на русском

**Обоснование**: Единый языковой стандарт упрощает работу русскоязычной команды и обеспечивает однозначное понимание требований.

### V. Качество и проверочные врата

Каждое изменение ДОЛЖНО проходить автоматизированные проверочные врата перед слиянием:

- Линтинг без ошибок и предупреждений
- Все тесты проходят успешно
- Форматирование кода соответствует стандарту проекта
- Код-ревью минимум одним разработчиком (кроме автора)
- Отсутствие TODO-заглушек критической функциональности в мёрж-реквестах

**Обоснование**: Автоматизированные врата предотвращают деградацию качества и обеспечивают соблюдение принципов конституции.

## Требования к технологиям

- Технологический стек выбирается исходя из задачи, ДОЛЖЕН поддерживать современный стандарт (Principle I)
- Зависимости управляются через стандартный менеджер пакетов стека
- Версии зависимостей фиксируются (lock-файлы обязательны)
- Сторонние библиотеки ДОЛЖНЫ быть активно поддерживаемыми (последний релиз ≤ 12 месяцев назад)
- Инфраструктура и конфигурация ДОЛЖНЫ быть воспроизводимыми (docker-compose, dev-контейнеры или эквивалент)

## Рабочий процесс разработки

- Каждая новая функция разрабатывается в отдельной ветке
- Ветки и коммиты следуют соглашению об именовании (например, `feature/описание`, `fix/описание`)
- Pull Request ДОЛЖЕН содержать описание изменений, ссылку на задачу и результаты тестирования
- Слияние в основную ветку только после прохождения всех проверочных врат (Principle V)
- Релизы версионируются по SemVer (MAJOR.MINOR.PATCH)

## Governance

Данная конституция имеет приоритет над всеми остальными практиками и соглашениями в проекте.

- **Внесение изменений**: Любая поправка в конституцию ДОЛЖНА быть оформлена как PR с явным описанием причин и последствий.
- **Версионирование**: Версия конституции следует SemVer:
  - MAJOR — удаление или несовместимое переопределение принципов
  - MINOR — добавление новых принципов или разделов, расширение трактовок
  - PATCH — уточнения формулировок, исправления опечаток
- **Проверка соблюдения**: Все PR ДОЛЖНЫ проверяться на соответствие принципам конституции ревьюером.
- **Периодический пересмотр**: Конституция ДОЛЖНА пересматриваться не реже одного раза в квартал.
